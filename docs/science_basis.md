# Science Basis of neurophase — cognitive safety and adaptive performance loop

*Теоретичне обґрунтування архітектури `neurophase` через три нейрокогнітивні механізми 2026 року: cognitive surrender, executive function under stress, та cognitive processing speed.*

---

## 1) Теоретичне обґрунтування: чому neurophase потрібен саме зараз

`neurophase` виходить з тези, що в системах людина+AI головний ризик — не лише помилка моделі, а **деградація когнітивного контуру користувача**. Це проявляється у трьох механізмах:

1. **Cognitive surrender** (Wharton, 2026): користувачі схильні приймати AI-відповіді без достатньої критичної перевірки, особливо коли відповідь виглядає впевненою.
2. **Executive function under stress** (Ming, 2026): під стресом просідають робоча памʼять, інгібіторний контроль і метакогнітивний моніторинг; це знижує якість рішень у високотемпових середовищах.
3. **Cognitive processing speed** (NIH, 2026): цілеспрямоване тренування швидкості обробки інформації може покращувати довгострокову когнітивну стійкість і зберігати продуктивність.

Отже, neurophase має діяти не як «автопілот», а як **оркестратор самоорганізації**: система налаштовує умови (таймінг, пороги, фідбек), за яких людина зберігає виконавчий контроль, а AI підсилює мислення замість його заміщення.

---

## 2) Мапінг механізмів на компоненти системи

### A. `R(t)` gate → prevention of cognitive surrender

**Механізм.** Якщо cognitive surrender зростає, користувач швидше погоджується з AI без верифікації.

**Компонент.** `R(t)` gate працює як динамічний дозвільний механізм дії:
- блокує або сповільнює виконання при низькій когнітивній готовності;
- примушує пройти короткий verification step (контрприклад, друге джерело, sanity check);
- повертає користувача в active-reasoning режим.

**Реалізація в коді.** `neurophase.gate.execution_gate.GateDecision` інваріант `I₁`: при `R(t) < θ` execution_allowed **не може** бути True (перевіряється в `__post_init__`). Модуль `neurophase.state.executive_monitor` додатково надає `VerificationStep` — structured friction перед high-impact дією.

**Теоретичний ефект.** `R(t)` gate зменшує ймовірність «пасивного прийняття» через структурований friction у критичних точках рішення.

---

### B. EEG beta → executive function monitor

**Механізм.** За stress-induced overload погіршується executive function (інгібіція імпульсивних дій, тримання цілі, переключення уваги).

**Компонент.** Потужність і динаміка **EEG beta** використовуються як онлайн-маркер стану когнітивного контролю:
- beta-ознаки комбінуються з контекстом задачі (темп, складність, помилки);
- при ознаках перевантаження система знижує темп, скорочує когнітивну ширину інтерфейсу, додає guided prompts;
- при стабільному контролі повертає нормальну автономію.

**Реалізація в коді.** `neurophase.state.executive_monitor.ExecutiveMonitor` обчислює `OverloadIndex` з beta-power trajectory, HRV та error-burst контексту, та видає `PacingDirective` (NORMAL / SLOW_DOWN / HARD_BLOCK).

**Теоретичний ефект.** Замість реакції «після помилки» система переходить до **превентивного executive monitoring**.

---

### C. Session archive → cognitive training loop

**Механізм.** Processing speed та резилієнс покращуються, коли є повторюване тренування з адаптивною складністю і зворотним звʼязком.

**Компонент.** `session archive` зберігає часові ряди рішень, помилок, latency, втручань гейта і recovery-патернів:
- формує індивідуальний профіль когнітивного темпу;
- генерує персоналізовані мікро-вправи на швидкість обробки та перевірку гіпотез;
- відстежує тренд переносу навичок у реальні сесії.

**Теоретичний ефект.** Архів перетворює разові інтервенції на **замкнений цикл навчання**, де система адаптується під користувача, а користувач тренує власний контроль.

## 3) Falsifiable predictions (перевірювані прогнози)

### Prediction 1 — for cognitive surrender / `R(t)` gate

**Гіпотеза.** Увімкнений `R(t)` gate з verification step зменшує частку некритично прийнятих AI-помилок.

**Операціоналізація.**
- Дизайн: A/B (gate ON vs gate OFF).
- Метрика: `False-Accept Rate` = частка завідомо хибних AI-відповідей, прийнятих без виправлення.
- Критерій фальсифікації: якщо протягом N сесій `ΔFAR` між ON/OFF статистично не відрізняється або гірший для ON, гіпотеза відхиляється.

---

### Prediction 2 — for executive function / EEG beta monitor

**Гіпотеза.** Beta-індекс перевантаження передбачає виконавчі збої раніше, ніж поведінкові помилки стають явними.

**Операціоналізація.**
- Дизайн: стрес-протокол з контрольованим підвищенням темпу/складності.
- Метрики: AUC/PR для прогнозу `error burst` у вікні 30–120 с наперед.
- Критерій фальсифікації: якщо модель на EEG beta не перевершує базову модель без EEG (latency + історія помилок), гіпотеза відхиляється.

---

### Prediction 3 — for processing speed / session archive training loop

**Гіпотеза.** Персоналізований цикл тренування через `session archive` прискорює когнітивну обробку без втрати точності.

**Операціоналізація.**
- Дизайн: longitudinal (мінімум 6–8 тижнів), adaptive-training vs static-practice.
- Метрики: `decision latency`, `accuracy under time pressure`, `recovery time after overload`.
- Критерій фальсифікації: якщо немає значущого покращення latency або покращення досягається лише ціною падіння точності, гіпотеза відхиляється.

---

## 4) Що це покращує в проєкті прямо зараз

- Зміщує neurophase з «ще одного AI-асистента» до **cognitive integrity system**.
- Дає чітку привʼязку між наукою і кодовими вузлами (`R(t)`, EEG monitor, session archive).
- Робить roadmap експериментально чесним: кожне ключове твердження має критерій спростування.
- Відкриває шлях до метрик продукту, які важливіші за «субʼєктивно зручно»: зниження false-accept, раннє виявлення overload, прискорення без втрати якості.

## 5) Practical implementation notes (next)

1. Додати в execution pipeline обовʼязковий `verification micro-step`, який активується `R(t)` при ризику surrender.
2. Винести beta-монітор у модуль стану (`state/executive_monitor`) з API для threshold/pacing control.
3. Розширити `session archive` до формату, придатного для longitudinal analytics (timestamped interventions + outcomes).
4. Зафіксувати preregistered evaluation protocol для трьох прогнозів, щоб уникнути post-hoc інтерпретацій.
